Descripción y Alcance del Proyecto

01 Gestión de Socios

Registro en línea de asociados con validaciones obligatorias (DNI único, domicilio, teléfono), automatizando la incorporación al Libro de Registro digital.

02 Módulo Transaccional

Historial completo de pagos de cuotas sociales mensuales y control de membresía con pasarela de pago recurrente mediante Mercado Pago.

03 Recaudación Transparente

Visualización pública de campañas de recaudación (insumos, obras y aparatología) con barras de progreso financiero calculadas en tiempo real.

04 Auditoría de Donaciones

Checkout interactivo para registrar transferencias bancarias manuales mediante carga de comprobantes, y donación directa en línea.

05 Panel Administrativo

Interfaz de control exclusiva para operadores para aprobar/rechazar transferencias, dar de alta/baja campañas y gestionar el padrón de socios.

06 Módulo Informativo

Portal dinámico para la publicación y difusión de novedades, obras concretadas y logros institucionales de la cooperadora del hospital.

Persistencia Híbrida y Concurrencia

Arquitectura de Datos y Estrategia de Persistencia

SQL Transaccional ACID

Consistencia Financiera

  • `usuarios`: Credenciales con contraseñas hasheadas en `bcryptjs`.
  • `perfiles_socios`: Registro legal de asociados del hospital (DNI único).
  • `campanas_eco`: Balances, metas de recaudación y control de saldos.

Justificación: Datos estructurados críticos que requieren atomicidad, consistencia relacional e integridad transaccional estricta.

NoSQL Documental Flexible

Flexibilidad y Multimedia

  • `noticias_actualidad`: Artículos con arrays de imágenes y videos dinámicos.
  • `campanas_detalle`: Narrativa larga, testimonios y estados de obras enlazados.

Justificación: Ideal para almacenamiento jerárquico sin esquemas rígidos, agilizando la carga de galerías multimedia.

⚡ Temas Clave de Exposición para el TFI:

1. Data Mashup Sincrónico (Sequelize + Mongoose): El backend unifica las consultas paralelas mediante Promise.all() para obtener datos de SQL y NoSQL en una sola llamada REST, unificándolas en un JSON plano. Esto reduce la latencia de red eliminando peticiones redundantes desde el cliente.
2. Prevención de Condiciones de Carrera: Las donaciones concurrentes en `donar` inician una transacción relacional que ejecuta un bloqueo exclusivo a nivel de fila (SELECT ... FOR UPDATE). Esto encola peticiones simultáneas, evitando que dos hilos actualicen el balance a la vez con valores desactualizados.

Patrones de Diseño y Refactorización

Análisis de Código y Propuestas de Refactorización

Abstracción e Interfaz
Pasarela de Pagos Genérica

Módulo: socioSubscriptionController.js y mpService.js
Solución: Crear una interfaz/contrato abstracto (`IPaymentGateway`) con firmas (`procesarPagoRecurrente()`, `cancelarSuscripcion()`) para desacoplar el SDK de Mercado Pago.
Ventaja: Desacoplamiento total. Permite migrar a Stripe, MODO u otros sin modificar controladores.
Desventaja: Node.js/JS no tiene interfaces nativas; debe emularse estructuralmente.

class IPaymentGateway { procesarPagoRecurrente() { ... } }
Factory Method (Creacional)
Creación de Usuarios

Módulo: services/userService.js
Solución: Utilizar una clase fábrica que, dado el rol (Socio o Admin), devuelva la instancia correcta del modelo con permisos específicos precargados.
Ventaja: Centraliza la lógica de instanciación y cumple el principio Open/Closed al extender roles.
Desventaja: Agrega sobreingeniería innecesaria si la jerarquía de roles es estática.

UserFactory.createUser(role, data);
Singleton (Creacional)
Conexión de BD Única

Módulo: backend/config/db.js (PostgreSQL & MongoDB).
Solución: Implementar el patrón Singleton para asegurar que toda la aplicación de Express comparta y reutilice una única instancia activa de conexión para ambos motores.
Ventaja: Ahorro óptimo de memoria y previene la saturación del pool de conexiones.
Desventaja: Introduce estado global, dificultando mocks en tests unitarios.

class DB { static getInstance() { ... } }
State Pattern + Async Worker
Conciliación de Transferencias

Módulo: donacionController.js (Aprobación de comprobantes).
Solución: Delegar estados de donación a clases (`EstadoPendiente`, `EstadoAprobado`, `EstadoRechazado`) con un Worker/Cron en segundo plano de conciliación asíncrona automatizada.
Ventaja: Elimina el cuello de botella de aprobación manual del operador; actualizaciones financieras inmediatas.
Desventaja: Mayor complejidad técnica por el Cron/Worker secundario.

donation.transitionTo(new ApprovedState());
